NagaraNetra — AI-Driven Parking Intelligence & Enforcement Platform
Flipkart GridLock Hackathon · Round 2 · Prototype Submission

====================================================================
1. THE PROBLEM
====================================================================
On-street illegal parking and spillover parking near commercial areas, metro
stations and events choke carriageways and intersections. Enforcement today is
PATROL-BASED and REACTIVE — officers go where they already expect violations,
which reinforces existing patrol habits instead of revealing where the real
problem is. There is no heatmap of parking violations versus their congestion
impact, and it is hard to prioritise which zones to police.

Problem direction: How can AI-driven parking intelligence detect illegal parking
hotspots and QUANTIFY their impact on traffic flow to enable TARGETED enforcement?

NagaraNetra answers this end-to-end on ~298,450 real Bengaluru Traffic Police
violation records (54 police stations, 09 Nov 2023 – 08 Apr 2024) across three
objectives: DETECT where/when parking clusters, QUANTIFY how much each cluster
actually hurts traffic, and PREDICT tomorrow's hotspots so patrols are
pre-positioned instead of dispatched after the fact.

====================================================================
2. CORE ARCHITECTURE — "BUILD ONCE, QUERY ALWAYS"
====================================================================
All heavy computation runs OFFLINE, once, in batch pipelines and is persisted to
SQLite. The Streamlit dashboard is read-only and never runs clustering or ML at
session time — so every tab loads in sub-second time on indexed SQLite reads.

    clean_parking_data.csv (~298K rows)
            |
            v
    build_hotspots.py        <- offline: 17 independent HDBSCAN runs
    build_forecast.py        <- offline: trains + predicts next-day hotspots
            |
            v
    hotspots.db / forecast.db / enrichment.db  (precomputed SQLite stores)
            |
            v
    app.py (Streamlit dashboard)  <- reads SQLite only, zero runtime ML

Three separate databases by design so they survive each other's rebuilds:
  - hotspots.db    — violations + 3 hotspot layers + metadata (9 indexes)
  - enrichment.db  — cached OSM context + TomTom live-traffic snapshots
  - forecast.db    — next-day predictions + model-vs-baseline metrics

A real correctness fix lives in the pipeline: the raw CSV's hour/weekday/date
were derived from created_datetime in UTC, but Bengaluru is UTC+5:30. Every
temporal field is re-derived in IST (Asia/Kolkata) before bucketing, so the time
buckets — and the dashboard's "current IST -> context" logic — are not misaligned
by 5.5 hours. A late-UTC event correctly rolls to the next IST date.

====================================================================
3. OBJECTIVE 1 — MULTI-CONTEXT HOTSPOT DETECTION (17 HDBSCAN RUNS)
====================================================================
We run HDBSCAN (Hierarchical Density-Based Spatial Clustering with Noise) on the
geographic coordinates — metric = haversine (true great-circle distance on
lat/long in radians), cluster_selection_method = "eom" (excess of mass, stable
boundaries). HDBSCAN needs no preset number of zones, finds clusters of arbitrary
shape and uneven density (a dense CBD and a sparse suburb are both found with the
same parameters), and explicitly labels sparse points as noise (-1) so we never
patrol non-problems. We build THREE layers — 1 + 10 + 6 = 17 independent runs:

  LAYER 1 — Persistent (1 run, min_cluster_size 30)
     Full dataset -> 1,992 stable, all-time chronic zones, ranked by total count.

  LAYER 2 — Contextual (10 runs, min_cluster_size 15)
     One HDBSCAN per (day_type x time_bucket): weekday/weekend x
     night / morning_peak / midday / evening_peak / late_night.
     -> 5,386 clusters, ranked by an enforcement score. Powers Live Enforcement.

  LAYER 3 — Monthly (6 runs, min_cluster_size 20)
     One HDBSCAN per calendar month (Nov 2023 .. Apr 2024) -> 4,177 clusters,
     ranked by monthly count. Shows how hotspot geography emerges/shifts over time.

WHY 17 SEPARATE RUNS BEAT ONE GLOBAL HDBSCAN + "FILTER BY TIME"
A single global clustering computes its density estimate, cluster boundaries AND
noise labels from the WHOLE-YEAR point distribution. If you then just filter that
one clustering by time, you are only subsetting points inside FIXED zones whose
geometry was frozen by the annual density. That approach silently assumes:

        Hotspot(location, Time A) == Hotspot(location, Time B)

But parking density is itself context-dependent. An office-district spot that is
dense on weekday mornings is empty on weekend nights; a restaurant/nightlife
cluster of ~18 points on a weekend night is a real, actionable hotspot that would
never survive a global min_cluster_size=30 threshold computed over a whole year —
so a single-model-then-filter system is BLIND to it. By re-running HDBSCAN
independently per context, the density estimate, the cluster boundaries and the
noise classification are all re-derived from only that context's points, so:

  - cluster geometry is ALLOWED to change with context (Hotspot(loc,A) != Hotspot(loc,B)
    when the data supports it) — we TEST the assumption instead of accepting it;
  - min_cluster_size is tuned per layer (30 / 15 / 20) for the density each scope has;
  - NEW zones that only emerge in a context appear, and zones that vanish drop out;
  - each layer is ranked by the signal that matters for it (all-time volume,
    enforcement score, monthly volume).

This is a more mathematically rigorous representation of parking behaviour, and it
is the foundation for time-sensitive, targeted enforcement.

Storage: each layer is one table; contextual rows are separated by a `context`
column (PK (context, cluster_id)) and monthly rows by `year_month`
(PK (year_month, cluster_id)). Cluster IDs are intentionally NOT global — each run
numbers from 0, meaningful only paired with its scope key — because each layer is
always queried in isolation via a composite index (single B-tree lookup, zero
runtime computation).

4. OBJECTIVE 2 — QUANTIFYING TRAFFIC IMPACT (ENRICHED ENFORCEMENT QUEUE)
====================================================================
Counting tickets is not the same as measuring congestion. We build the enforcement
queue in two stages.

STAGE A — Enforcement Score (violation pressure, no external data)
For every contextual cluster:
    enforcement_score = 0.6 x percentile(all-time volume in this context)
                      + 0.4 x percentile(recent volume, last 30 days)
60% captures the chronic, underlying pattern; 40% boosts zones that are getting
worse right now — momentum — without needing any external feed. This produces the
first ranked Enforcement Priority Queue.

STAGE B — Traffic Impact Re-Ranking (the hard part the problem asks for)
On demand (button-triggered), each top zone's CENTROID (the mean lat/long of the
violations inside the cluster — never the police-station location) is enriched and
re-scored 0–100, then RE-RANKED with a Δ-rank arrow (up/down vs the volume-only
order). We fuse:

  - ROAD CRITICALITY (OpenStreetMap, 50 m around centroid): highway class ->
    importance 1–5 (motorway/trunk=5 ... residential=1), lane count, one-way,
    segment length. A violation on a trunk arterial blocks far more flow than one
    on a residential lane. road_criticality = 0.6*importance + 0.4*lanes_norm.

  - URBAN ACTIVITY (OpenStreetMap, 1 km around centroid): counts of hospitals,
    offices, schools/colleges/universities, transit (bus stops, railway stations),
    malls/markets and places of worship — i.e. how much legitimate parking demand
    the area generates. Parking lots count NEGATIVELY (legitimate supply offsets
    demand). Weighted sum, min-max normalised across the current top-20.

  - VEHICLE SEVERITY: dominant vehicle's carriageway footprint
    (scooter/motorcycle=1 ... bus=5, tanker/lorry/HGV=6) / 6.

  - JUNCTION RISK: 1 if the cluster sits at a real junction (blocks turns and
    sightlines), else 0.

  - LIVE CONGESTION (TomTom, optional): congestion_index = 1 - current/free-flow
    speed at the centroid.

IMPACT SCORE (weights sum to 1.0; live component folds out when no key is set):
    WITH live  = 100 x (0.30 enforcement + 0.18 road + 0.18 urban
                        + 0.14 congestion + 0.12 vehicle + 0.08 junction)
    WITHOUT    = 100 x (0.35 enforcement + 0.20 road + 0.20 urban
                        + 0.15 vehicle + 0.10 junction)

Every zone also gets a plain-English "Why" reason built from its top factors
(e.g. "Trunk road · near 15 hospitals · transit hub") — the explainable part of
the ranking, so a supervisor understands WHY a zone jumped the queue.

HONEST HANDLING OF TOMTOM (why live traffic is kept aside)
TomTom returns CURRENT, real-time road speeds. Our dataset is HISTORICAL
(Nov 2023 – Apr 2024), so live congestion CANNOT be matched one-to-one to past
violations — today's traffic did not cause a 2024 ticket. We therefore treat the
live layer honestly: it is optional, fetched fresh (never cached), clearly labelled
"demo only" in the UI, and FOLDS OUT of the formula entirely when no key is set, so
the platform runs fully and correctly on the keyless 5-component formula. In a
real-time deployment the same code path streams live and the impact score would
react instantly to actual on-road congestion. This is a deliberate scientific
decision, not a missing feature.

Resilience: OSM context is static, so it is cached permanently in enrichment.db,
deduped to an ~11 m grid key so the public Overpass API is never hit twice per
centroid. Every external call degrades gracefully to neutral defaults / estimates,
so a table or map ALWAYS renders even if an API is down.

5. OBJECTIVE 3 — PREDICTIVE FORECASTING ("TOMORROW'S HOTSPOTS")
====================================================================
The hotspot layers are descriptive (what already happened). The forecast layer is
PROACTIVE — it directly attacks the "patrol-based and reactive" pain point.

  - HONEST TARGET: predicting an individual next event's exact lat/long is
    irreducible noise and not learnable. What IS learnable and operationally useful
    is the EXPECTED zone-day intensity: how many violations each zone will see
    tomorrow. We forecast that and rank zones.
  - DAY, NOT HOUR (data-driven): profiling showed (zone x hour) cells average < 1
    violation (too sparse); (zone x day) averages ~4.2 when active — real signal.
  - DATA: a dense (zone x day) panel with structural zeros from a zone's first
    appearance to the last date (~280K panel rows over 1,992 zones).
  - FEATURES: calendar + cyclical (weekday, is_weekend, day-of-year sin/cos,
    is_holiday for the India/Karnataka calendar), lag & rolling history
    (lag_1/7/14, rolling mean 7/14/30, rolling std, expanding mean) — every one
    shift(1)-ed within the zone to GUARANTEE no leakage — plus zone-static context
    and optional cached OSM.
  - MODEL: scikit-learn HistGradientBoostingRegressor(loss="poisson") — the
    statistically correct objective for non-negative, zero-inflated counts with
    mean-dependent variance; one GLOBAL gradient-boosted model over the panel
    shares strength across zones, handles mixed numeric+categorical features and
    NaN lags natively, and trains in seconds (no new heavy dependency — sklearn
    already ships via hdbscan). Two quantile models give a coherent q10–q90
    confidence band. Benchmarked against a strong seasonal-naive baseline.
  - EVALUATION (30-day temporal holdout, train on the past, score the unseen
    future): the model BEATS the baseline on EVERY metric —
        MAE 1.02 vs 1.11 (-8.4%), Poisson deviance 2.37 vs 3.93 (-40%),
        Precision@10 0.23 vs 0.18, NDCG@10 0.41 vs 0.31, Spearman 0.30 vs 0.21.
    The ranking metrics (which decide patrol prioritisation) improve the most.
    Example: a zone surging 93 -> 106 over two days is read through lag_1 /
    roll_mean_7 and forecast at ~64, catching an escalation the baseline is blind to.

====================================================================
6. THE ACTION LAYER — FROM INSIGHT TO DEPLOYMENT
====================================================================
A shared action engine turns the ranked hotspots into decision-ready output that
all three operational views read from one source of truth:

  COMMAND CENTER  — "What should we do right now?" Priority cards per zone with
     severity (HIGH/MEDIUM/LOW), traffic-impact and forecast-risk scores, a "Why
     this location?" factor list, the recommended action, and (after dispatch) the
     nearest-unit ETA. Auto-follows the current IST day/time, with a manual override.

  SMART BRIEFINGS — deterministic, offline, template-based natural-language shift
     summaries per zone (no LLM call: instant and fully reproducible). One-click
     "Generate All" produces a downloadable, dated shift-briefing .txt for handover.

  PATROL DISPATCH (ROUTE TRACKING) — "Who should go where?" Given selected station
     units (and ad-hoc on-road units), it computes a unit x hotspot travel-time
     MATRIX via TomTom Matrix Routing v2, assigns the nearest free unit to the
     highest-priority hotspot first (greedy), then pulls TomTom Calculate Route for
     the FINAL assigned pairs to draw the actual, traffic-aware route polyline + ETA
     + distance on the map. Every network call degrades gracefully to a haversine
     estimate (~25 km/h urban speed, 1.3 detour factor) shown as a dashed route, so
     dispatch always works — with the UI clearly flagging live vs estimated.

7. ENGINEERING QUALITY
====================================================================
  - Two-phase design: heavy compute offline + cached; dashboard is read-only and
    sub-second on indexed SQLite reads (9 indexes; composite (context, rank)).
  - Read-only query layer is cached across reruns (st.cache_data) so a 298K-row
    table is not re-queried on every interaction.
  - Three separate SQLite stores so rebuilding one never wipes another.
  - Graceful degradation everywhere: missing DBs show friendly notices; every OSM /
    TomTom call falls back to neutral defaults or estimates — the app never breaks.
  - Pinned light theme + responsive maps for consistent presentation on any screen.

TECH STACK: Python · Streamlit · Folium / streamlit-folium (maps) · HDBSCAN
(clustering) · scikit-learn (Poisson GBM forecasting) · pandas / numpy / scipy ·
SQLite (precomputed stores) · requests. APIs: OpenStreetMap Overpass (keyless) and
TomTom Traffic + Routing (optional key — app runs fully without it).

8. FUTURE SCOPE
====================================================================
  - AGENTIC AI ENFORCEMENT COPILOT: today's intelligence is powered by classical
    ML — HDBSCAN (unsupervised clustering) and a Poisson gradient-boosted forecaster
    — and briefings are deterministic templates (instant, reproducible, offline).
    The next leap is a GenAI agent (e.g. a Claude-powered copilot) layered on top
    that (a) writes richer, context-aware shift narratives and answers free-text
    questions over the hotspot/forecast data ("which weekend-evening zones near
    hospitals are trending up this week?"), and (b) acts AGENTICALLY — continuously
    monitoring the live forecast + congestion feeds and autonomously proposing (and,
    with human-in-the-loop approval, triggering) patrol re-assignments, escalations
    and station notifications. This turns NagaraNetra from a decision-SUPPORT dashboard
    into a decision-MAKING copilot, while the ML layer keeps every recommendation
    grounded, explainable and auditable.
  - AUTOMATED STATION DISPATCH: email/notify each police station's inbox with its
    zone-specific shift briefing and assigned patrol route at the start of every
    shift — closing the loop from detection to deployment without manual handover.
  - IN-HOTSPOT MICRO-TARGETING: instead of only the cluster centroid, use the
    individual violation lat/longs INSIDE a hotspot to compute a kernel-density /
    sub-cluster "heat core" — pinpointing the exact kerb segment within the zone
    where violations concentrate, and an optimised patrol path through those points,
    so an officer is sent to the precise spot, not just the general area.
  - REAL-TIME DEPLOYMENT: stream live TomTom congestion on a 2–5 min cadence so the
    impact score reacts to actual on-road conditions (removing the historical-vs-live
    date mismatch entirely).
  - REPEAT-OFFENDER RADAR and JURISDICTION-AWARE LABELLING: surface recurring
    vehicles, and split border-straddling clusters across the top-2 police-station
    jurisdictions instead of showing only the dominant station.
  - LONGER-HORIZON FORECASTING: as more history accrues, extend from next-day to
    multi-day / weekly intensity forecasts.

9. WHY THIS WINS
====================================================================
  - Real data, real scale — ~298K actual Bengaluru violations, not synthetic.
  - Unsupervised & label-free — HDBSCAN finds organic hotspots and discards noise,
    re-derived per context so geography adapts to time instead of being assumed.
  - Beyond a heatmap — it does the problem's hard part: QUANTIFYING congestion
    impact by fusing violations + road criticality + urban demand + live traffic
